efficient branching using git-flow by Vincent Driessen

About

git-flow are a set of git extensions to provide high-level repository operations for Vincent Driessen's branching model. more

★ ★ ★

This cheatsheet shows the basic usage and effect of git-flow operations

★ ★ ★

Basic tips

  • Git flow provides excellent command line help and output. Read it carefully to see what's happening...
  • The macOS/Windows Client Sourcetree is an excellent git gui and provides git-flow support
  • Git-flow is a merge based solution. It doesn't rebase feature branches.

★ ★ ★

Setup

  • You need a working git installation as prerequisite.
  • Git flow works on macOS, Linux and Windows

★ ★ ★

macOS

Homebrew
$ brew install git-flow-avh
Macports
$ port install git-flow-avh

Linux

$ apt-get install git-flow

For detailed git flow installation instructions please visit the git flow wiki.

Getting started

Git flow needs to be initialized in order to customize your project setup.

★ ★ ★

Initialize

Start using git-flow by initializing it inside an existing git repository:

git flow init

You'll have to answer a few questions regarding the naming conventions for your branches.
It's recommended to use the default values.

Features

  • Develop new features for upcoming releases

★ ★ ★

Start a new feature

Development of new features starting from the 'develop' branch.

Start developing a new feature with

git flow feature start my-feature-name

This action creates a new feature branch based on 'develop' and switches to it

Publish the feature

Push a feature to GitHub so it can be used by other users.

git flow feature publish my-feature-name

Make commits

When working on your feature, make commits to this branch. Push them to GitHub.

git add -p
git commit
git push

Pull request review

Have your teammates review your code for accuracy and quality.

  • Merge target is 'develop'
  • Assign to everyone who can review.
  • Link the PR in your project's Slack channel.

If you have gh installed, here's a quick way to create a PR:

gh pr create --base develop

Finish the feature

Finish the development of a feature after being reviewed.

  • Merge my-feature-name into 'develop'
  • Use Github PR buttons. Do not use 'git flow feature finish'
  • Branch should be deleted from remote (enable in github project settings)
Click Merge Pull Request in GitHub

Working on a published feature

Get a feature pushed to GitHub by another user.

git flow feature pull origin my-feature-name

Make a release

  • Prepare a new release
  • Supports full blown features and minor bug fixes
  • Helps in Preparing meta-data for a release

Start a release

To start a release, use the git flow release command. It creates a release branch from the 'develop' branch.

Typically, we should be using semver versioning. We may optionally include labels in the tag that correspond to app store builds or anything else that makes our lives easier.

git flow release start v9.1.21

You can optionally set a commit hash to start the release from any commit on 'develop'. This is useful to build a release from any point in develop's history.

git flow release start v9.1.21 [HASH]

Publish the release

Push the release branch to GitHub to allow finalizing commits by others.

git flow release publish v9.1.21

Finish up a release

Finishing a release is one of the big steps in git branching. It performs several actions:

  • Merges the release branch back into 'main'
  • Tags the release with its name
  • Back-merges the release into 'develop'
  • Removes the release branch
git flow release finish v9.1.21

Don't forget to push your tags.

git push origin --tags

Hotfixes

  • Use feature branches instead, if possible.
  • Hotfixes arise from the necessity to act immediately upon an undesired state of a live production version
  • May be branched off from the corresponding tag on the main branch that marks the production version.

★ ★ ★

Start a hotfix

Like the other git flow commands, a hotfix is started with

git flow hotfix start HOTFIX_NAME [RELEASE]

Publish a hotfix

You're seeing the pattern here right? Push the hotfix to GitHub.

git flow hotfix publish HOTFIX_NAME

Finish a hotfix

By finishing a hotfix it gets merged back into develop AND main. Additionally the main merge is tagged with the hotfix version.

git flow hotfix finish HOTFIX_NAME
